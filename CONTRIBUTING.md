# Contributing to ArbiterAI

Thanks for your interest in contributing.

## Before You Start

- Check existing issues before opening a new one.
- For larger changes, open a proposal issue first.
- Keep pull requests focused on one topic.

## Workflow

1. Fork the repository and create a branch.
2. Implement your change with tests or examples when relevant.
3. Update documentation when behavior or APIs change.
4. Open a pull request with a clear description.

## Branch Naming

Use one of the following branch prefixes:

- `feature/<short-description>`
- `fix/<short-description>`
- `docs/<short-description>`
- `chore/<short-description>`

## Commit Messages

Use concise, imperative commit messages.

Examples:

- `Add policy middleware pipeline validation`
- `Fix Git MCP tool registration in builder`
- `Update README package architecture table`

## Pull Request Checklist

- [ ] Change is scoped and focused.
- [ ] Public API changes are documented.
- [ ] Tests or usage examples are included when applicable.
- [ ] README or docs were updated if needed.
- [ ] No unrelated refactors are bundled.

## Reporting Bugs

When reporting bugs, include:

- environment details (.NET version, OS)
- reproduction steps
- expected behavior
- actual behavior
- logs or stack traces if available

## Security Issues

Do not open public issues for sensitive vulnerabilities.

Please report security concerns privately to the maintainers once a security contact channel is published.
